## New Conflicts Check (Rev 2 Changes)

### Rev 2 Additions Reviewed

The revision log (line 8) documents substantial changes: reframed `/dependency-audit` as CLI scanner coordinator, added composability model, removed inline secrets patterns, scoped `/compliance-check`, added Security Maturity Levels, split into 3 phases, added prompt injection countermeasures, added secret redaction rules, added `--security-override` flag, added `attribution` field requirement.

**No new blocking conflicts found.** Detailed findings:

1. **Security Maturity Levels and settings.json** -- The plan proposes reading `security_maturity` from `.claude/settings.json` or `.claude/settings.local.json` (line 311). CLAUDE.md documents that project-level overrides use these files and that "Lists merge -- project settings layer on top of global settings." The proposed storage location is consistent. No conflict.

2. **`/ship` delegation to `/secrets-scan` via Task subagent** -- Deviation #3 (line 960) establishes a new pattern: a skill dispatching a Task subagent to execute another deployed skill. This is novel but consistent with the existing Task tool usage in `/ship` and `/dream`. CLAUDE.md's Tool Permissions table includes `Task` as pre-authorized. No conflict.

3. **`/audit` composability with `/secure-review`** -- Deviation #4 (line 962) has `/audit` delegating its security scan to `/secure-review` when deployed. This is backward-compatible (falls back to built-in scan when not deployed). Consistent with the Glob-then-delegate pattern used by `/ship` for agent discovery. No conflict.

4. **`based_on` field in `redhat-security.json`** -- Changed from `extends` (round 1) to `based_on` with explicit "(reference only, not runtime inheritance)" annotation. Deviation #2 (line 958) documents this clearly. Resolves the round 1 concern. No conflict.

5. **3-phase rollout** -- The split into Phases A/B/C with separate plan files is consistent with CLAUDE.md's skill development workflow (edit, validate, deploy). Each phase is independently `/dream`-able and `/ship`-able. No conflict.

6. **`--security-override` flag** -- Follows the `$ARGUMENTS` scope parameter pattern (CLAUDE.md Pattern #9). The plan documents parsing from `$ARGUMENTS` (line 868). Consistent.

7. **`attribution` field for `threat-model-gate`** -- The plan requires `attribution: Original work, claude-devkit project` in frontmatter (line 205). This matches the pattern established by `receiving-code-review` (Reference archetype). No conflict.

8. **INCOMPLETE verdict for `/dependency-audit`** -- A new verdict value (`INCOMPLETE`) not in the standard PASS/PASS_WITH_NOTES/BLOCKED scale. CLAUDE.md documents verdict gates as "PASS/FAIL/BLOCKED" (Pattern #4). This is a minor deviation but well-justified: a scanner-unavailable state is semantically distinct from PASS (nothing wrong found) and BLOCKED (something wrong found). The plan explicitly states the skill "must NOT report PASS" and "must NOT fall back to LLM-based CVE guessing" (line 177). This is a defensible design choice. Not a blocking conflict.

## Optional Suggestions

1. **Registry table model format consistency.** The plan's proposed registry entries (lines 531-537) use full model identifiers (`claude-opus-4-6`, `claude-sonnet-4-5`), while CLAUDE.md's existing registry uses abbreviated forms (`opus-4-6`). When updating CLAUDE.md in Phase C Step C2, consider normalizing the entire Model column to one format. This is a Phase C implementation detail, not a plan conflict.

2. **INCOMPLETE verdict documentation.** The `/dependency-audit` skill introduces a new verdict value (`INCOMPLETE`) outside the standard PASS/PASS_WITH_NOTES/BLOCKED scale. Consider adding `INCOMPLETE` to the CLAUDE.md "Verdict gates" pattern description in the Phase C CLAUDE.md update, so future skill authors know it is an acceptable verdict for scanner-dependent skills.

3. **`security_maturity` field location.** The plan proposes reading the maturity level from `.claude/settings.json`. Consider documenting whether this field belongs in the global `~/.claude/settings.json` or only in project-level `.claude/settings.json`. The current text says "`.claude/settings.json` or `.claude/settings.local.json`" which implies project-level, but a team might want to set a global default. This is an implementation-time decision, not a plan issue.
