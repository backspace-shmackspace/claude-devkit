# Plan: /retro Skill and /ship Integration

## Context

Claude Devkit skills generate rich review artifacts during `/ship` runs -- code reviews, QA reports, and test failure logs. These artifacts are archived to `plans/archive/<name>/` and never consulted again. This represents a significant information loss: recurring patterns (e.g., "coders consistently forget to add index exports," "reviewers catch missing error boundaries in every review") are re-discovered from scratch each run rather than learned once and applied going forward.

This plan introduces two capabilities:

1. **`/retro` standalone skill** (Scan archetype) -- mines archived review artifacts, extracts recurring patterns, categorizes learnings by actor (coder/reviewer/QA/tester), and writes them to a persistent project-level memory file.
2. **`/ship` Step 7 integration** -- a lightweight post-commit step that runs a retro subagent on the just-archived artifacts and appends learnings inline, so retrospective capture happens automatically without requiring a separate `/retro` invocation.

**Current state:** No learning mechanism exists. Each `/ship` run starts with zero institutional knowledge about what previous runs discovered.

**Constraints:**
- Must follow v2.0.0 skill patterns (11 architectural patterns)
- `/retro` must be a Scan archetype (parallel analysis, severity-rated findings, verdict gate)
- Ship integration must be non-blocking (Step 7 failures must not affect the commit)
- Learnings file must be human-readable, git-trackable, and machine-parseable
- Deduplication required -- same learning must not appear twice
- Staleness management -- learnings must have dates and be purgeable

## Goals

1. Create a `/retro` skill that extracts actionable learnings from review artifact history
2. Integrate a lightweight retro step into `/ship` as Step 7 (post-commit, non-blocking)
3. Define the `.claude/learnings.md` schema with deduplication and staleness semantics
4. Enable agents (coders, reviewers, QA) to consume learnings during `/ship` runs
5. Categorize findings by actor calibration: what coders miss vs. what reviewers catch vs. what tests surface

## Non-Goals

1. **Real-time agent prompt injection** -- Learnings are read by agents at step start, not injected mid-execution
2. **Cross-project learning** -- Learnings are project-scoped only (no global memory store)
3. **Automated learning pruning** -- Staleness dates are written; manual pruning is left to the user or a future `/retro prune` mode
4. **Weighting or confidence scoring** -- Learnings are binary (present or absent), not scored
5. **Changing existing agent base templates** -- Agent templates in `templates/agents/` are not modified; learnings consumption is done via the `/ship` skill prompts

## Assumptions

1. `plans/archive/` contains at least 2-3 feature archives to mine meaningfully (the repository currently has 9 archived features with code review artifacts)
2. Archived code-review and QA report files exist within feature archive directories but may not follow a uniform naming convention (inner filenames may not match directory names)
3. The `.claude/` directory exists in target projects (created by Claude Code on first use)
4. Users want learnings to be git-tracked (committed alongside code) rather than ephemeral
5. The `claude-sonnet-4-5` model is sufficient for pattern extraction (no opus required for mining)

## Proposed Design

### Component 1: `.claude/learnings.md` Schema

The learnings file is the persistent memory store. It lives at `.claude/learnings.md` in each project, is git-tracked, and has a structured format that is both human-readable and machine-parseable.

**Schema:**

```markdown
# Project Learnings

> Auto-generated by /retro and /ship. Edit manually to curate.
> Last updated: 2026-03-12T10:00:00-04:00

## Coder Patterns

Patterns observed in coder agent behavior. Feed these back when dispatching coders.

### Missed by coders, caught by reviewers

- **[2026-03-01] Missing index exports** -- Coders create new modules but forget to add re-exports in barrel index files. Seen in: feature-x, feature-y. #coder #export #structure
- **[2026-03-05] Error boundary gaps** -- Components are created without error boundaries. Reviewers catch this in every React feature. Seen in: add-dashboard, add-settings. #coder #react #error-handling

### Missed by coders, caught by tests

- **[2026-02-28] Async cleanup missing** -- useEffect hooks lack cleanup functions, causing test warnings. Seen in: add-notifications. #coder #react #testing

## Reviewer Patterns

Patterns observed in code reviewer behavior.

### Consistently caught (high-value reviewer checks)

- **[2026-03-01] SQL injection in dynamic queries** -- Reviewer reliably flags string interpolation in SQL. Seen in: add-search, add-filters. #reviewer #security #sql

### Overcorrected (reviewer false positives)

- **[2026-03-05] Premature optimization flags** -- Reviewer flags O(n) loops in handlers processing <100 items. These are not performance issues. Seen in: add-tags. #reviewer #performance #false-positive

## QA Patterns

Patterns observed in QA validation.

### Recurring test failures

- **[2026-03-01] Missing test fixtures** -- Tests fail because fixture files referenced in test plan are not created. Seen in: add-export. #qa #fixtures #setup

### Coverage gaps

- **[2026-03-05] Edge case: empty state** -- QA consistently finds missing empty-state handling. Seen in: add-dashboard, add-list-view. #qa #edge-case #ux

## Test Patterns

Patterns observed in automated test results.

### Flaky tests

- **[2026-03-01] Timer-dependent tests** -- Tests using setTimeout/setInterval flake in CI. Seen in: add-polling. #test #flaky #timing

### Common failures

- **[2026-03-05] Import path case sensitivity** -- Tests pass on macOS but fail in CI (Linux) due to case-insensitive imports. Seen in: add-utils. #test #ci #imports
```

**Key design decisions:**

| Decision | Rationale |
|----------|-----------|
| Markdown format | Human-readable, git-diffable, no tooling dependency |
| Date prefix `[YYYY-MM-DD]` | Enables staleness detection (>90 days = stale) |
| `Seen in:` suffix | Deduplication anchor -- if same pattern seen in new feature, update `Seen in:` list instead of adding new entry |
| Hash tags | Machine-parseable categorization without formal schema |
| Section hierarchy | Actor-based grouping (Coder/Reviewer/QA/Test) matches how agents consume learnings |
| Bold learning title | Scannable by humans and agents |

**Deduplication guidance:**

When adding a new learning:
1. Read all existing entries in the target section
2. If an existing learning describes the same underlying issue (same root cause, same actor, same category), update the existing entry: append the new feature name to the `Seen in:` list and update the date to the most recent occurrence
3. If no existing learning matches the underlying issue, append as a new entry under the appropriate section
4. Err on the side of creating new entries -- duplicates are easier to clean up than incorrectly merged patterns

**Staleness semantics:**

- Each entry has a date `[YYYY-MM-DD]` representing the most recent occurrence
- Entries older than 90 days are considered stale (not automatically removed, but `/retro` will flag them)
- Users can manually delete stale entries

### Component 2: `/retro` Standalone Skill (Scan Archetype)

**Archetype:** Scan (parallel analysis, synthesis, verdict gate)
**Model:** `claude-opus-4-6` (coordinator), `claude-sonnet-4-5` (scan subagents)
**Steps:** 6

**Workflow:**

```
Step 0 -- Determine scope (recent/full/<feature-name>)
Step 1 -- Scan: Coder calibration analysis (parallel)
Step 2 -- Scan: Reviewer calibration analysis (parallel)
Step 3 -- Scan: Test pattern analysis (parallel)
Step 4 -- Synthesis and deduplication
Step 5 -- Write learnings and verdict gate
```

**Scope modes:**
- `recent` (default): Last 3 archived features by git commit date
- `full`: All archived features
- `<feature-name>`: Single specific feature

**Input artifacts (per feature archive, discovered via glob):**
- `plans/archive/<name>/*.code-review.md`
- `plans/archive/<name>/*.qa-report.md`
- `plans/archive/<name>/*.test-failure.log` (if exists)
- `plans/archive/<name>/*.feasibility.md` (if exists)

**Output artifacts:**
- `plans/retro-[timestamp].coder-scan.md` -- Coder pattern analysis
- `plans/retro-[timestamp].reviewer-scan.md` -- Reviewer pattern analysis
- `plans/retro-[timestamp].test-scan.md` -- Test pattern analysis
- `plans/retro-[timestamp].summary.md` -- Synthesized summary with verdict
- `.claude/learnings.md` -- Updated learnings file (created if missing)

### Component 3: `/ship` Step 7 Integration

**Position:** After Step 6 (commit gate), only on PASS/PASS_WITH_NOTES
**Model:** `claude-sonnet-4-5` (single Task subagent)
**Blocking:** No -- Step 7 failures are logged but do not affect the commit outcome

**Behavior:**
1. Read the just-archived artifacts from `plans/archive/[name]/`
2. Read existing `.claude/learnings.md` (if exists)
3. Extract learnings from this single feature's review cycle
4. Deduplicate against existing learnings
5. Append new learnings to `.claude/learnings.md`
6. Auto-commit `.claude/learnings.md` with message `chore: update project learnings from /ship run`
7. Report what was learned

**Ship prompt changes for learnings consumption (Steps 3c, 4a, 4c):**
- Coder dispatch (Step 3c): Add to prompt -- "If `.claude/learnings.md` exists, read the `## Coder Patterns` section first. Apply any relevant learnings to your implementation."
- Code review (Step 4a): Add to prompt -- "If `.claude/learnings.md` exists, read the `## Reviewer Patterns` section. Verify past coder mistakes are not repeated."
- QA validation (Step 4c): Add to prompt -- "If `.claude/learnings.md` exists, read the `## QA Patterns` section. Check for recurring coverage gaps."

**Detailed implementation of `skills/retro/SKILL.md`:**

```markdown
---
name: retro
description: Mine review artifacts for recurring patterns and write project learnings.
version: 1.0.0
model: claude-opus-4-6
---
# /retro Workflow

## Inputs
- Scope: $ARGUMENTS (optional: "recent", "full", or a feature name)
  - `recent`: Last 3 archived features (default)
  - `full`: All archived features
  - `<name>`: Single feature archive (e.g., "add-user-auth")

## Role
You are the **retro coordinator**. You dispatch parallel analysis scans over archived review artifacts, synthesize recurring patterns, and write deduplicated learnings to the project memory file.
You do NOT fix code or modify agents — you extract and record patterns.

## Step 0 — Determine scope and discover artifacts

Tool: `Bash`, `Glob` (direct — coordinator does this)

**Determine scope:**
- If `$ARGUMENTS` is empty: scope = "recent"
- Else: scope = `$ARGUMENTS`

Validate scope:
- If scope is "recent", "full", or matches an existing directory in `plans/archive/`: proceed
- Else: stop with "Invalid scope. Use: /retro [recent|full|<feature-name>]"

Derive timestamp: `[timestamp]` = current ISO datetime (e.g., `2026-03-12T10-00-00`)

**Discover artifacts:**

Tool: `Bash`

```bash
ARCHIVE_DIR="./plans/archive"

if [ "$SCOPE" = "recent" ]; then
  # Get 3 most recently added feature directories (by git commit date, excluding sync/ and audit/)
  FEATURES=$(git log --diff-filter=A --name-only --format='' -- "$ARCHIVE_DIR"/*/ \
    | grep -E '^plans/archive/[^/]+/$' \
    | grep -v '/sync/' | grep -v '/audit/' \
    | sed 's|plans/archive/||;s|/||' \
    | awk '!seen[$0]++' \
    | head -3)
elif [ "$SCOPE" = "full" ]; then
  FEATURES=$(ls -d "$ARCHIVE_DIR"/*/ 2>/dev/null | grep -v '/sync/$' | grep -v '/audit/$' | xargs -I{} basename {})
else
  # Single feature
  if [ -d "$ARCHIVE_DIR/$SCOPE" ]; then
    FEATURES="$SCOPE"
  else
    echo "No archive found for feature: $SCOPE"
    exit 1
  fi
fi

echo "Features to analyze: $FEATURES"
FEATURE_COUNT=$(echo "$FEATURES" | wc -w | tr -d ' ')
echo "Total features: $FEATURE_COUNT"
```

**Fail fast if no artifacts:**
- If `$FEATURE_COUNT` is 0: stop with "No archived features found in plans/archive/. Run /ship on at least one feature first."

**Collect artifact paths (glob-based discovery):**

For each feature in `$FEATURES`, use glob to discover artifacts:
- `plans/archive/<feature>/*.code-review.md` (all code review files in the directory)
- `plans/archive/<feature>/*.qa-report.md` (all QA report files in the directory)
- `plans/archive/<feature>/*.test-failure.log` (all test failure logs in the directory)

Log any feature directories that contain no code-review or QA artifacts: "Skipping <feature>: no review artifacts found."

Remove features with zero artifacts from the scan set. Re-check `$FEATURE_COUNT` after filtering.

Store the feature list and discovered artifact paths for use in Steps 1-3.

## Step 1 — Scan: Coder calibration (parallel with Steps 2, 3)

Tool: `Task`, `subagent_type=general-purpose`, `model=claude-sonnet-4-5`

Prompt:
"You are analyzing code review artifacts to identify coder behavior patterns.

Read the following code review files:
[list all discovered *.code-review.md paths from Step 0]

Read each code review file in its entirety. Extract findings regardless of the specific section header format used. Look for issues categorized by severity (critical, major, minor) or described as problems, concerns, or areas for improvement. Also look for positive feedback or things done well, regardless of what section header they appear under.

For each code review, extract:
1. **Critical/Major findings** — These represent things the coder missed. Rate each: Critical / High / Medium / Low.
2. **Positives** — These represent things the coder did well
3. **Feature name** — The feature being reviewed (use the archive directory name)

Then analyze across all reviews to identify:

### Recurring coder mistakes (caught by reviewers)
Patterns that appear in 2+ reviews. For each:
- Pattern name (concise, descriptive)
- Severity: Critical / High / Medium / Low
- Description (what the coder keeps missing)
- Features where this occurred (Seen in: list)
- Category tags (#coder #<topic>)

### Coder strengths (consistently done well)
Patterns that appear as Positives in 2+ reviews.

### One-off issues (appeared only once)
List but mark as non-recurring. Include severity rating.

Write to `./plans/retro-[timestamp].coder-scan.md` with this structure:

```markdown
# Coder Calibration Scan — [timestamp]

## Recurring Mistakes (caught by reviewers)
- **[Pattern Name]** [Severity] — [Description]. Seen in: [feature1, feature2]. #coder #tag
...

## Coder Strengths
- **[Pattern Name]** — [Description]. Seen in: [feature1, feature2]. #coder #tag
...

## One-Off Issues
- **[Issue]** [Severity] — [Description]. Seen in: [feature]. #coder #tag
...

## Statistics
- Features analyzed: N
- Code reviews found: N
- Recurring patterns: N
- One-off issues: N
```"

Issues in QA reports that the code reviewer did not flag. Rate severity.

## Step 4 — Synthesis and deduplication

Tool: `Read` (direct — coordinator does this)

Read all three scan reports:
- `./plans/retro-[timestamp].coder-scan.md`
- `./plans/retro-[timestamp].reviewer-scan.md`
- `./plans/retro-[timestamp].test-scan.md`

Read existing learnings (if any):
- `.claude/learnings.md` (may not exist)

**Synthesize:**

For each recurring pattern from the scan reports:
1. Read all existing entries in the target section of `.claude/learnings.md`
2. If an existing learning describes the same underlying issue (same root cause, same actor, same category): mark for update (append new `Seen in:` features, update date)
3. If no existing learning matches the underlying issue: mark for addition under the appropriate section
4. Err on the side of creating new entries -- duplicates are easier to clean up than incorrectly merged patterns

**Categorize into learnings sections:**

| Scan Source | Learning Section |
|-------------|-----------------|
| Coder recurring mistakes | `## Coder Patterns > ### Missed by coders, caught by reviewers` |
| Coder strengths | Not written to learnings (positive reinforcement is implicit) |
| Reviewer high-value checks | `## Reviewer Patterns > ### Consistently caught` |
| Reviewer overcorrections | `## Reviewer Patterns > ### Overcorrected` |
| Missed by reviewer, caught by QA | `## QA Patterns > ### Coverage gaps` |
| Recurring test failures | `## Test Patterns > ### Common failures` |
| Recurring coverage gaps | `## QA Patterns > ### Coverage gaps` |
| Test infrastructure issues | `## Test Patterns > ### Flaky tests` or `### Common failures` |

**Generate summary:**

Write `./plans/retro-[timestamp].summary.md`:

```markdown
# Retro Summary — [scope] — [timestamp]

## Verdict
[LEARNINGS_FOUND / NO_NEW_LEARNINGS / INSUFFICIENT_DATA]

## New Learnings
[Count: N]
- [Learning 1 — brief description]
- [Learning 2 — brief description]
...

## Updated Learnings (existing patterns seen again)
[Count: N]
- [Learning 1 — added features to Seen in list]
...

## Stale Learnings (>90 days since last occurrence)
[Count: N]
- [Learning 1 — last seen YYYY-MM-DD]
...

## Statistics
- Features analyzed: N
- Code reviews mined: N
- QA reports mined: N
- Test failure logs mined: N
- Total recurring patterns found: N
- New learnings added: N
- Existing learnings updated: N
- Stale learnings flagged: N

## Reports
- Coder scan: ./plans/retro-[timestamp].coder-scan.md
- Reviewer scan: ./plans/retro-[timestamp].reviewer-scan.md
- Test scan: ./plans/retro-[timestamp].test-scan.md
```

**Verdict rules:**
- **LEARNINGS_FOUND**: At least 1 new or updated learning
- **NO_NEW_LEARNINGS**: All patterns already exist in learnings.md and no updates needed
- **INSUFFICIENT_DATA**: Fewer than 2 features analyzed AND no existing `.claude/learnings.md` to cross-reference against (cannot identify recurring patterns and cannot update existing ones)

Note: Single-feature mode (`/retro <feature-name>`) can still produce LEARNINGS_FOUND if it finds patterns that match existing learnings in `.claude/learnings.md` (updating `Seen in:` lists counts as an update). It only returns INSUFFICIENT_DATA when there are no existing learnings to cross-reference against.

"New learnings written to .claude/learnings.md:

### Scan Archetype Alignment

The `/retro` skill matches the Scan archetype from `templates/skill-scan.md.template`:

### Deviations from Standard Patterns

| Deviation | Justification |
|-----------|---------------|
| Verdict values differ from standard (LEARNINGS_FOUND vs PASS) | Standard PASS/FAIL/BLOCKED don't semantically fit a learning-extraction workflow. The retro never "fails" -- it either finds learnings or doesn't. INSUFFICIENT_DATA maps to the "cannot proceed meaningfully" semantic of BLOCKED. |
| Writes to `.claude/learnings.md` (not just `./plans/`) | Learnings are a persistent project resource consumed by agents, not a one-time report. `.claude/` is the correct location for agent-consumed project configuration. |
| Ship Step 7 is non-blocking (no verdict gate) | This is an explicit design choice: the commit (Step 6) is the true gate. Learning capture is best-effort and must never regress the core /ship workflow. |
| No bounded iteration language in retro | Retro is single-pass analysis with no revision loop. Bounded iteration pattern does not apply. |
| Ship Step 7 is single-pass with no bounded iteration | Step 7 is non-blocking and best-effort. A revision loop would add latency to a post-commit step that must not delay the user. Bounded iteration pattern does not apply. |
