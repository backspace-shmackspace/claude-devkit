### Commit 0eb9e0a: Security Guardrails Phase B

This commit embedded security guardrails into three core skills:

**1. /ship v3.4.0 → v3.5.0:**
- Added **three security maturity levels** (L1 advisory, L2 enforced, L3 audited) controlled via `.claude/settings.json` or `.claude/settings.local.json` with `security_maturity` field
- Added **three security gates**:
  - **Secrets scan** (Step 0 pre-flight): Runs /secrets-scan on working directory. BLOCKS at all maturity levels (committed secrets cannot be un-committed). Override available with `--security-override`.
  - **Secure review** (Step 4d verification): Runs /secure-review on uncommitted changes. At L1: BLOCKED auto-downgrades to PASS_WITH_NOTES (prominent warning). At L2/L3: BLOCKED stops workflow unless overridden.
  - **Dependency audit** (Step 6 commit gate): Runs /dependency-audit on manifest files. At L1: BLOCKED auto-downgrades to PASS_WITH_NOTES. At L2/L3: BLOCKED stops workflow unless overridden.
- Added **--security-override flag** with reason string: `/ship path/to/plan.md --security-override "reason"`
  - Logs override reason for audit trails (especially at L3)
  - Downgrades BLOCKED to PASS_WITH_NOTES and continues workflow
- At L2/L3, checks that all three security skills are deployed (secrets-scan, secure-review, dependency-audit) in Step 0 pre-flight. Fails fast if missing.
- Security gates are **conditional** — only run if the corresponding skill is deployed. At L1, missing skills log warnings. At L2/L3, missing skills block pre-flight.

**2. /architect v3.0.0 → v3.1.0:**
- Added **threat-model-gate detection** in Step 0: Globs for `~/.claude/skills/threat-model-gate/SKILL.md`
- Added **security-sensitive feature detection** in Step 2: Uses keyword heuristics (auth, password, token, encrypt, pii, upload, payment, etc.) to identify security-related features
- **If both conditions are met** (threat-model-gate deployed AND feature is security-sensitive):
  - Injects a "SECURITY CONTEXT" block into the architect Task prompt
  - Requires architect to include a `## Security Requirements` section with:
    - Assets at risk (data classification)
    - Trust boundaries
    - STRIDE analysis (Spoofing, Tampering, Repudiation, Information Disclosure, DoS, Elevation of Privilege)
    - Proposed mitigations
  - References the threat-model-gate skill for the full checklist
- **If threat-model-gate is NOT deployed or feature is NOT security-sensitive:** Standard planning prompt only (no injection)

**3. /audit v3.0.0 → v3.1.0:**
- Added **secure-review composability** in Step 2: Globs for `~/.claude/skills/secure-review/SKILL.md`
- **If found AND scope is NOT "plan":** Invokes /secure-review instead of the built-in security scan
  - Maps audit scopes: 'code' → 'changes', 'full' → 'full'
  - Writes findings to `./plans/audit-[timestamp].security.md` (uses audit naming convention, not secure-review convention, for synthesis compatibility)
  - Skips the built-in security scan (Step 2 Task delegation)
- **If NOT found OR scope is "plan":** Uses built-in security scan (unchanged behavior)
  - /secure-review scans code, not plans, so plan audits always use built-in scan

### Why Security Maturity Levels Matter

The three-level model provides **progressive security enforcement**:
- **L1 (advisory)**: Default for all projects. Security scans run and report, but don't block delivery. Suitable for early-stage projects, prototypes, or teams ramping up security practices.
- **L2 (enforced)**: Security BLOCKED verdicts stop the workflow. Override available for false positives or time-sensitive situations (with logged reason). Suitable for production codebases with security requirements.
- **L3 (audited)**: Same as L2, but all overrides are logged for compliance/audit trails. Suitable for regulated environments (FedRAMP, HIPAA, SOC 2, PCI-DSS).

The model is **non-breaking** (L1 is the default) and **composable** (security skills are conditionally invoked only if deployed).
