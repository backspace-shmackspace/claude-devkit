## Recurring Coverage Gaps

- **Validator not executed at QA time** [High] — The `python3 generators/validate_skill.py` command was listed as an acceptance criterion but marked NOT VERIFIED in both the audit-remove-mcp-deps and sync-remove-mcp-deps QA reports. The QA agent relied on static inspection rather than running the tool, deferring the runtime gate to a post-QA Phase 3 step. This means a structural failure in the validator (wrong exit code, unexpected output) could have merged undetected. Seen in: audit-remove-mcp-deps, sync-remove-mcp-deps. #qa #coverage #validator

- **Integration / end-to-end tests not executed** [High] — Live skill invocation (`/audit code`, `/sync recent`) was listed as a blocking gate in both MCP-removal rollout plans but was deferred out of the QA report with "no automated mechanism exists to verify this in static review." No QA report shows evidence of a live smoke test being run in any of the three features. Seen in: audit-remove-mcp-deps, sync-remove-mcp-deps, receiving-code-review (behavioral smoke test skipped as "optional"). #qa #coverage #integration

- **Full test suite skipped** [Medium] — `bash generators/test_skill_generator.sh` was specified as acceptance criterion #18 in the receiving-code-review report and was explicitly NOT VERIFIED because it was "a long-running integration suite." The phase0-reference-validator report is the only one where the full suite (33 tests) was actually run to completion. Seen in: receiving-code-review, audit-remove-mcp-deps, sync-remove-mcp-deps. #qa #coverage #test-suite

- **`--strict` mode not tested for new skill types** [Low] — Two separate QA reports noted that `--strict` mode validation was not explicitly tested: receiving-code-review noted it was not tested but inferred it would pass (0 errors, 0 warnings), and phase0-reference-validator noted no strict-mode test exists for Reference skills. If strict-mode behavior diverges from non-strict for new archetypes, it would go undetected. Seen in: receiving-code-review, phase0-reference-validator. #qa #coverage #strict-mode

- **Unknown/invalid `type` field values not tested** [Low] — The phase0-reference-validator report flagged that a Reference skill with `type: unknown` emits only a warning rather than an error, and this path has no dedicated test. A similar gap exists implicitly for the receiving-code-review skill whose `type: reference` value was accepted by the validator but the rejection path for a bad type value was never exercised. Seen in: phase0-reference-validator, receiving-code-review. #qa #coverage #edge-case

## Test Infrastructure Issues

- **No automated gate between QA and merge** [High] — Across all three features, runtime checks (validator execution, integration smoke tests, full test suite) are described as "deferred to Phase 3" or "non-blocking." There is no CI mechanism that enforces these gates before merge. The QA agent cannot run them and documents them as open items instead. This means the burden falls entirely on the developer to remember to close them. Seen in: audit-remove-mcp-deps, sync-remove-mcp-deps, receiving-code-review. #test #infra #ci

- **Test suite has a numbering gap (test 26 missing)** [Low] — The phase0-reference-validator QA report documents that the test suite jumps from Test 25 to Test 27, with no Test 26. Test 27b was added as a sub-test to maintain the total count of 33. The count is correct but the non-sequential numbering makes it harder to reason about coverage from the suite output alone. Seen in: phase0-reference-validator. #test #infra #numbering

- **No test fixtures for edge-case CLI arguments to deploy.sh** [Low] — The phase0-reference-validator report notes that `--undeploy` with an empty string argument is not tested, relying on `set -euo pipefail` and argument count checks as implicit protection. This is a configuration-level gap: the protective mechanism exists but is not verified by the test suite. Seen in: phase0-reference-validator. #test #infra #fixture
