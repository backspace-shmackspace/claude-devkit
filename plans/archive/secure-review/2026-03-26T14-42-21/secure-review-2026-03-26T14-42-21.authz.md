### H-1. deploy_skill() and deploy_contrib_skill() lack input sanitization -- path traversal possible

**File:** `scripts/deploy.sh:19-31` (deploy_skill), `scripts/deploy.sh:34-47` (deploy_contrib_skill)
**STRIDE:** Tampering, Elevation of Privilege
**CWE:** CWE-22 (Improper Limitation of a Pathname to a Restricted Directory)
**OWASP:** A01 - Broken Access Control

**Description:** The `deploy_skill()` and `deploy_contrib_skill()` functions accept a skill name argument and construct filesystem paths by concatenating it directly into `$SKILLS_DIR/$skill` and `$DEPLOY_DIR/$skill` without validating that the name does not contain path traversal characters (`..`, `/`). This is in contrast to `undeploy_skill()` on line 53, which explicitly rejects names containing `/`, `..`, or starting with `-`.

An attacker or a malicious skill name could cause `cp` to read from or write to unintended directories:

```bash
# deploy_skill constructs paths without sanitization:
local src="$SKILLS_DIR/$skill"    # no validation
local dst="$DEPLOY_DIR/$skill"    # no validation
mkdir -p "$dst"
cp "$src/SKILL.md" "$dst/SKILL.md"
```

The `deploy_all_core()` function iterates directory entries (line 81: `skill="$(basename "$skill_dir")"`), which is safe because `basename` strips path components. However, the direct `deploy_skill "$1"` call at line 190 passes raw user input.

**Impact:** An attacker who controls the skill name argument (e.g., via a crafted plan or shell alias) could cause the deployment script to read SKILL.md from or write to arbitrary directories relative to `$SKILLS_DIR` or `$DEPLOY_DIR`.

**DREAD:** Damage=6, Reproducibility=8, Exploitability=5, Affected=3, Discoverability=6. Score: 5.6

**Remediation:** Add the same sanitization from `undeploy_skill()` to `deploy_skill()` and `deploy_contrib_skill()`:

```bash
deploy_skill() {
    local skill="$1"
    # Input sanitization: reject path traversal and flag-like names
    if [[ "$skill" == */* ]] || [[ "$skill" == *..* ]] || [[ "$skill" == -* ]]; then
        echo "ERROR: Invalid skill name: '$skill'" >&2
        return 1
    fi
    ...
}
```

### H-2. generate_senior_architect.sh uses unsanitized user input in sed substitution -- command injection via project type

**File:** `generators/generate_senior_architect.sh:297-298`
**STRIDE:** Tampering, Elevation of Privilege
**CWE:** CWE-78 (Improper Neutralization of Special Elements used in an OS Command)
**OWASP:** A03 - Injection

**Description:** The bash script uses `sed -i` to substitute `PROJECT_NAME` and `PROJECT_TYPE` placeholders with user-supplied values, without escaping sed special characters:

```bash
sed -i.bak "s/PROJECT_NAME/$PROJECT_NAME/g" "$AGENT_FILE"
sed -i.bak "s/PROJECT_TYPE/$PROJECT_TYPE/g" "$AGENT_FILE"
```

If `$PROJECT_TYPE` contains sed metacharacters (e.g., `/`, `&`, `\`), the substitution will fail or produce corrupted output. More critically, if the value contains a newline followed by an `e` sed command (on GNU sed), it could lead to command execution. On macOS BSD sed this is less exploitable but still causes incorrect file generation.

The Python equivalent (`generate_senior_architect.py`) uses `str.format()` which is immune to this class of injection.

**Impact:** A maliciously crafted `--project-type` argument could corrupt the generated agent file or, on GNU sed, potentially execute arbitrary commands.

**DREAD:** Damage=7, Reproducibility=7, Exploitability=5, Affected=2, Discoverability=5. Score: 5.2

**Remediation:** Escape sed metacharacters before substitution, or replace sed with a safe substitution mechanism:

```bash
# Escape sed special characters
SAFE_PROJECT_NAME=$(printf '%s\n' "$PROJECT_NAME" | sed 's/[&/\]/\\&/g')
SAFE_PROJECT_TYPE=$(printf '%s\n' "$PROJECT_TYPE" | sed 's/[&/\]/\\&/g')
sed -i.bak "s/PROJECT_NAME/$SAFE_PROJECT_NAME/g" "$AGENT_FILE"
sed -i.bak "s/PROJECT_TYPE/$SAFE_PROJECT_TYPE/g" "$AGENT_FILE"
```

Or preferably, migrate all users to the Python generator (`generate_senior_architect.py`) which does not have this vulnerability, and deprecate the bash script.

## Medium Findings

### M-1. Overly broad tool permission allowlist in .claude/settings.local.json -- privilege creep risk

**File:** `.claude/settings.local.json:1-39`
**STRIDE:** Elevation of Privilege
**CWE:** CWE-269 (Improper Privilege Management)
**OWASP:** A01 - Broken Access Control

**Description:** The local settings file contains a broad set of pre-authorized tool patterns including:

- `Bash(git:*)` -- authorizes ALL git commands without restriction
- `Bash(git commit:*)` -- pre-authorizes commits (could be used to commit sensitive files)
- `Bash(ssh:*)` -- authorizes SSH connections to any host
- `Bash(ssh-add:*)` -- authorizes loading SSH keys
- `Bash(pip3 install:*)` -- authorizes installing arbitrary Python packages
- `Bash(brew install:*)` -- authorizes installing system packages
- `Bash(python3:*)` -- authorizes executing arbitrary Python code
- `Bash(gh auth:*)`, `Bash(gh api:*)` -- authorizes GitHub API operations

While these are project-local (not global), they reduce the security boundary between Claude Code and the local system. The `Bash(python3:*)` permission is particularly broad as it allows execution of arbitrary Python scripts.

**Impact:** A compromised or misbehaving skill/agent could leverage these pre-authorized patterns to perform privileged operations (install packages, execute Python, SSH to hosts, modify git config) without user confirmation.

**DREAD:** Damage=5, Reproducibility=3, Exploitability=3, Affected=2, Discoverability=4. Score: 3.4

**Remediation:**
1. Narrow `Bash(git:*)` to specific subcommands (git status, git log, git diff, git add, git commit, git worktree).
2. Remove `Bash(python3:*)` -- replace with specific invocations like `Bash(python3 generators/*.py:*)`.
3. Remove `Bash(ssh:*)` unless actively needed, or scope to specific hosts.
4. Remove `Bash(pip3 install:*)` and `Bash(brew install:*)` -- these should prompt for user approval.
5. Document the security rationale for each remaining entry.

### M-3. Non-atomic file writes in generate_senior_architect.py and generate_agents.py

**File:** `generators/generate_senior_architect.py:350-351`, `generators/generate_agents.py:408-409`
**STRIDE:** Tampering
**CWE:** CWE-367 (Time-of-Check Time-of-Use Race Condition)
**OWASP:** A08 - Software and Data Integrity Failures

**Description:** Both generators write files directly without atomic write semantics:

```python
# generate_senior_architect.py line 350
with open(agent_file, 'w') as f:
    f.write(content)

# generate_agents.py line 408
with open(agent_file, 'w') as f:
    f.write(content)
```

In contrast, `generate_skill.py` uses proper atomic writes (temp file + `os.replace()`), which prevents partial file corruption if the process is interrupted.

**Impact:** If the write is interrupted (power loss, signal, disk full), the target file could be left in a corrupted state with partial content, potentially causing Claude Code to load a malformed agent definition.

**DREAD:** Damage=3, Reproducibility=2, Exploitability=2, Affected=2, Discoverability=3. Score: 2.4

**Remediation:** Use the same `atomic_write()` function from `generate_skill.py`, or use the temp file + rename pattern:

```python
import tempfile
fd, tmp_path = tempfile.mkstemp(dir=agent_dir, prefix=".agent-", suffix=".tmp")
with os.fdopen(fd, 'w') as f:
    f.write(content)
os.replace(tmp_path, agent_file)
```

### M-4. /ship skill uses `git add -A` in revision loop -- stages all files including potential secrets

**File:** `skills/ship/SKILL.md:533-534` (Step 5a)
**STRIDE:** Information Disclosure
**CWE:** CWE-200 (Exposure of Sensitive Information)
**OWASP:** A02 - Cryptographic Failures (secrets exposure)

**Description:** In Step 5a (revision loop pre-commit), the /ship skill uses `git add -A` to stage all files before creating a WIP commit:

```bash
git add -A
git commit -m "WIP: ship v3.3.0 first-pass implementation (pre-revision)"
```

This stages ALL files in the working directory, including any that may have been inadvertently created during the first-pass implementation (e.g., `.env` files, credential caches, debug dumps, core dumps). While the final commit in Step 6 uses explicit file lists from the plan, this WIP commit could include sensitive files that are later squashed into the final commit via `git reset --soft HEAD~N`.

**Impact:** Secrets or sensitive files created during implementation could be permanently committed to the git history, even after the squash, because the intermediate commit objects remain in the git reflog.

**DREAD:** Damage=5, Reproducibility=3, Exploitability=2, Affected=3, Discoverability=3. Score: 3.2

**Remediation:** Replace `git add -A` with explicit staging of only the plan's scoped files:

```bash
# Stage only files from the plan's task breakdown
git add <files-from-plan-scope>
git commit -m "WIP: ship v3.3.0 first-pass implementation (pre-revision)"
```

### M-5. /ship skill uses `git reset --soft HEAD~N` without branch protection check

**File:** `skills/ship/SKILL.md:583` (Step 6)
**STRIDE:** Tampering, Repudiation
**CWE:** CWE-284 (Improper Access Control)
**OWASP:** A01 - Broken Access Control

**Description:** Step 6 of /ship performs a `git reset --soft HEAD~N` to squash WIP commits. The skill does not verify that the current branch is not `main`/`master` before performing this destructive operation. While the skill instructions say "coordinator reads plan" and the plan was approved, there is no programmatic guard against running this on a protected branch.

The /dream skill (Step 5) has a detached HEAD check before committing, but /ship lacks an equivalent branch protection check before the reset.

**Impact:** If /ship is run on `main` directly (instead of a feature branch), the `git reset --soft HEAD~N` could rewrite history on a protected branch, potentially causing data loss if followed by a force push.

**DREAD:** Damage=6, Reproducibility=3, Exploitability=3, Affected=3, Discoverability=4. Score: 3.8

**Remediation:** Add a branch protection check before the reset:

```bash
CURRENT_BRANCH=$(git symbolic-ref --short HEAD 2>/dev/null || echo "DETACHED")
if [[ "$CURRENT_BRANCH" == "main" || "$CURRENT_BRANCH" == "master" ]]; then
    echo "WARNING: Running on protected branch '$CURRENT_BRANCH'. Skipping soft reset."
    # Fall back to creating a new commit without squashing
fi
```

## Trust Boundary Analysis

### Agent Delegation Trust Model

The codebase implements a multi-tier trust model through agent delegation:

1. **Skills (Tier 0)** -- Coordinator skills (`/dream`, `/ship`, `/audit`) run in the user's Claude Code session with full tool access as defined by `~/.claude/settings.json` and `.claude/settings.local.json`.

2. **Task Subagents (Tier 1)** -- Skills delegate work to Task subagents that run with the same tool permissions as the parent session. There is no permission reduction between the coordinator and the delegated agent. A coder subagent has the same filesystem access as the coordinator.

3. **Worktree Isolation (Structural)** -- The /ship skill uses git worktrees to provide structural isolation between parallel work groups. This is a file-scope boundary (which files an agent should modify), not a permission boundary (what tools the agent can use). Agents in worktrees still have full Read/Write/Bash access.

**Key observation:** There is no privilege escalation risk in the traditional sense because all agents run with the same permissions as the invoking user. The security model relies on:
- Claude Code's tool allowlist (what Bash commands are pre-authorized)
- Skill definitions (what agents are instructed to do)
- File boundary validation in /ship (what files agents are allowed to modify)

The file boundary validation in /ship Step 3d is a defense-in-depth measure, not an access control boundary.
