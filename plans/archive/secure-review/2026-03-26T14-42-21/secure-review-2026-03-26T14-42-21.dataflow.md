## High Findings

### H-1. `/secrets-scan` writes raw file contents (including potential secrets) to `./plans/` before scanning

**Files:**
- `skills/secrets-scan/SKILL.md:73-106` (Step 1)

**Description:**
The `/secrets-scan` skill collects target content into `./plans/secrets-scan-[TIMESTAMP].scan-target.txt` BEFORE running pattern detection. In `all` scope, this file contains the full verbatim content of every tracked file in the working directory (line 93: `cat "$f" >> "$SCAN_TARGET_FILE"`). In `staged` scope, it captures the full staged diff. If the codebase contains secrets, they are written to the plans directory in cleartext before being detected and redacted in Step 2.

The Step 5 archive operation moves this file to `./plans/archive/secrets-scan/[TIMESTAMP]/`, but it uses `|| true` (line 346), meaning archive failure is silently swallowed. If archival fails, the raw content persists in the plans directory.

**Data Flow:**
```
Source files (may contain secrets)
  -> cat into plans/secrets-scan-*.scan-target.txt (UNREDACTED)
  -> Pattern scan produces redacted findings (Step 2)
  -> Archive moves raw file (Step 5, may fail silently)
```

**Impact:** An observer with read access to `./plans/` during or after a scan could read actual secret values from the scan target file, even though the final report is properly redacted. If the archive step fails, the raw dump persists indefinitely.

**DREAD:** Damage=7, Reproducibility=9, Exploitability=4, Affected=6, Discoverability=5 => **6.2**

**Remediation:**
1. Write the scan target to a temporary file outside the plans directory (e.g., `/tmp/secrets-scan-*.tmp`) and delete it immediately after Step 2 completes, before the Task subagent in Step 3 runs.
2. Never archive the raw scan target file. Archive only the redacted findings.
3. If the file must remain for false-positive review, apply the same redaction patterns from Step 2 to the scan target file before writing it.

### H-3. Shell injection via unsanitized user input in `generate_senior_architect.sh`

**Files:**
- `generators/generate_senior_architect.sh:297-298`

**Description:**
The shell script passes user-supplied `PROJECT_TYPE` and `PROJECT_NAME` directly into `sed` substitution without any sanitization:

```bash
sed -i.bak "s/PROJECT_NAME/$PROJECT_NAME/g" "$AGENT_FILE"
sed -i.bak "s/PROJECT_TYPE/$PROJECT_TYPE/g" "$AGENT_FILE"
```

`PROJECT_NAME` is derived from `basename "$TARGET_DIR"` (line 53) and `PROJECT_TYPE` comes from interactive user input or CLI argument (lines 21, 49). If either value contains sed metacharacters (`/`, `&`, `\`), the sed command will break or inject unintended content. If a directory name or project type contains a `/`, the sed substitution will fail or produce corrupted output.

More critically, since these are double-quoted `sed` expressions, shell metacharacters in the variables could be interpreted by the shell before sed sees them.

The Python equivalent (`generate_senior_architect.py:344`) uses `str.format()` which is safe from shell injection but has its own (lower-risk) template injection concern with curly braces.

**Data Flow:**
```
User input (CLI arg or interactive prompt)
  -> $PROJECT_TYPE variable (unsanitized)
  -> sed substitution in double-quoted string
  -> Written to agent file
```

**Impact:** Corrupted agent file output; in adversarial scenarios, content injection into the generated agent definition. This is a local-only tool so exploitation requires the user to supply malicious input to their own tool, but it violates defense-in-depth principles.

**DREAD:** Damage=3, Reproducibility=7, Exploitability=5, Affected=2, Discoverability=6 => **4.6**

**Remediation:**
1. Sanitize `PROJECT_NAME` and `PROJECT_TYPE` before use in sed: strip or escape `/`, `&`, `\`, and newline characters.
2. Use a different sed delimiter: `sed -i.bak "s|PROJECT_TYPE|$PROJECT_TYPE|g"` reduces the `/` collision risk (but not `|`).
3. Prefer the Python generator (`generate_senior_architect.py`) which does not have this issue.
4. Consider deprecating the shell wrapper in favor of the Python version.

### M-3. `install.sh` reads `/etc/passwd` for shell detection

**Files:**
- `scripts/install.sh:29-30`

**Description:**
The install script falls back to reading `/etc/passwd` to detect the user's shell:

```bash
user_shell=$(getent passwd "$USER" 2>/dev/null | cut -d: -f7 || echo "")
```

While this is a standard Unix pattern and the output is only used for shell detection, it accesses a system file that on some configurations may contain additional user information. The `$USER` environment variable is read without sanitization, though exploitation here is limited to self-inflicted damage since the user runs this script locally.

**Impact:** Minimal in practice -- `/etc/passwd` is world-readable on Unix systems and the script only extracts the shell field. However, the pattern should be noted for completeness.

**DREAD:** Damage=1, Reproducibility=3, Exploitability=1, Affected=1, Discoverability=3 => **1.8**

### M-4. `generate_skill.py` path traversal allowlist includes `/tmp/` which is world-writable

**Files:**
- `generators/generate_skill.py:113-129`

**Description:**
The path traversal check explicitly allows target directories under `/tmp/`:

```python
tmp = Path("/tmp").resolve()
```

While `/tmp/` is useful for testing, allowing skill generation into a world-writable directory means:
- Another user on a shared system could pre-create a `/tmp/skills/` directory with a symlink attack.
- Generated skill files in `/tmp/` have default permissions visible to all users on the system.

The risk is low for single-user macOS systems but relevant for shared Linux development servers.

**Impact:** On multi-user systems, generated skills in `/tmp/` could be read or replaced by other users.

**DREAD:** Damage=3, Reproducibility=5, Exploitability=4, Affected=1, Discoverability=3 => **3.2**

**Remediation:**
1. Use `tempfile.mkdtemp()` with restrictive permissions instead of bare `/tmp/`.
2. Alternatively, restrict the `/tmp/` allowance to paths created by the generator itself (e.g., require the temp dir to match a specific prefix pattern).

### L-2. Bare `except:` clauses silently swallow errors

**Files:**
- `generators/generate_senior_architect.py:295` -- `except:`
- `generators/generate_skill.py:220` -- `except:`
- `generators/generate_skill.py:435` -- `except:`

**Description:**
Three bare `except:` clauses catch all exceptions (including `KeyboardInterrupt`, `SystemExit`) and silently continue. While not a direct data exposure issue, silent error swallowing can mask security-relevant failures (e.g., a permission error when trying to delete a temp file containing sensitive content).

At `generate_skill.py:220`, the bare except catches failures during temp file cleanup, meaning a temporary file containing generated skill content could persist on disk if `os.unlink()` fails silently.

**Impact:** Potential persistence of temporary files. No direct data exposure.

**DREAD:** Damage=2, Reproducibility=2, Exploitability=1, Affected=1, Discoverability=2 => **1.6**

**Remediation:**
Replace `except:` with `except Exception:` and add logging for cleanup failures.

### L-5. `deploy.sh` undeploy function uses `rm -rf` on user-derived path

**Files:**
- `scripts/deploy.sh:49-68`

**Description:**
The `undeploy_skill()` function constructs a path from user input and calls `rm -rf`:
```bash
local target="$DEPLOY_DIR/$skill"
rm -rf "$target"
```

The function includes input validation (lines 53-56) rejecting path traversal (`/`, `..`) and flag-like names (`-*`), which is good. However, the validation does not check for shell metacharacters, spaces, or symlinks at the target path.

The risk is mitigated by:
1. `$DEPLOY_DIR` is hardcoded to `$HOME/.claude/skills` (not user-controlled).
2. The skill name validation rejects `/` and `..`.
3. The comment at line 65-66 notes that `rm -rf` will trigger a Claude Code permission prompt.

**Impact:** Low -- validation covers the primary attack vectors. Edge case risk with symlinks.

**DREAD:** Damage=4, Reproducibility=2, Exploitability=2, Affected=1, Discoverability=3 => **2.4**

**Remediation:**
1. Add a symlink check before deletion: `if [ -L "$target" ]; then echo "ERROR: symlink detected"; return 1; fi`
2. Consider using `rm -rf --` to prevent accidental flag interpretation (already mitigated by the `-*` check).
