## Medium Findings
Count: 1

- **[Vulnerability] Unvalidated CLI argument interpolated into Python heredoc** (`scripts/audit-log-query.sh`, `cmd_trend` function, approx line 670)
  - The `--dimension` argument value is interpolated directly into a Python string literal inside a heredoc (`dimension = '${dimension}'`) without sanitization. A user supplying a crafted value like `--dimension "'; import os; os.system('evil'); #"` could break out of the Python string and execute arbitrary code.
  - **CWE-78** (Improper Neutralization of Special Elements used in an OS Command)
  - **Mitigating factors**: This is a local CLI tool run by the developer themselves (not network-facing). The attacker would need shell access, at which point they already have code execution. Exploitation requires deliberate crafting of a CLI argument.
  - **Recommendation**: Pass the dimension name via `sys.argv` instead of shell interpolation, consistent with how `compute-run-score.sh` and `score-reflector.sh` handle arguments. Example: `python3 - "$n" "$dimension" <<'PYEOF'` then read via `sys.argv[1]` and `sys.argv[2]`.

## Low Findings
Count: 2

- **[Vulnerability] Shell variable interpolated into Python triple-quoted string** (`scripts/audit-log-query.sh`, `cmd_scores` function, approx line 620)
  - The `score_event` variable (a JSONL line) is interpolated into a Python heredoc via `event_json = '''${score_event}'''`. If the JSONL line contains triple quotes (`'''`), this breaks the Python parser. Since JSONL is machine-generated by `emit-audit-event.sh` using `python3 json.dumps()`, the risk of malicious content is low. An attacker would need local filesystem write access to craft malicious audit log content.
  - **CWE-94** (Improper Control of Generation of Code)
  - **Recommendation**: Read the JSONL line via `sys.argv` or `sys.stdin` instead of shell interpolation into a Python string.

- **[Vulnerability] Stderr suppression may hide scoring failures** (`skills/ship/SKILL.md`, new lines in Step 6 commit gate)
  - `SCORE_JSON=$(bash scripts/compute-run-score.sh "$AUDIT_LOG" 2>/dev/null)` suppresses all stderr output from the scoring script. While the script is designed to always exit 0 and output neutral scores on error, suppressing stderr removes diagnostic warnings that could help debug scoring issues.
  - **Mitigating factors**: Score computation is non-blocking and informational. Failures do not affect the commit gate verdict.
  - **Recommendation**: Consider redirecting stderr to the audit log or a temporary file rather than `/dev/null`, so scoring diagnostics are available when debugging.
