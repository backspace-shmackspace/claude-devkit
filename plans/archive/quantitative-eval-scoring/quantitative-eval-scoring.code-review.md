## Major Improvements (Should Fix)

### M1: Shell variable interpolation of raw JSONL content into an inline Python string

**File:** `/Users/imurphy/projects/claude-devkit/scripts/audit-log-query.sh`, `cmd_scores()`, lines 609-643

The `score_event` variable (raw JSONL content extracted from a file) is interpolated directly into a double-quoted bash heredoc passed to `python3 -c "..."`:

```bash
score_event=$(grep '"event_type":"run_score"' "$log_file" 2>/dev/null | tail -1)
# ...
python3 -c "
event_json = '''${score_event}'''
```

Two risks, ordered by probability:

1. **Triple-single-quote delimiter break:** If `score_event` ever contained `'''`, the Python string literal would terminate early and cause a syntax error. Standard JSON cannot contain `'''`, but an adversarially crafted or manually edited JSONL file could. The script would fall back to `jq '.'` gracefully due to the `|| {}` handler, so the impact is a degraded output, not a crash.

2. **Bash special character expansion:** Shell characters (`$`, backtick, `\`) inside `${score_event}` are subject to bash expansion before Python sees them. `json.dumps()` in `emit-audit-event.sh` does not escape `$` signs in string values. The `details` field values generated by `compute-run-score.sh` currently contain no `$` signs (they are constructed from numeric counts and gate names), but this is not enforced by the schema. A `details` field containing `$(id)` or `$HOME` would be expanded by bash during command substitution.

The established pattern in this codebase for safely passing data to Python is to use a quoted heredoc (`<<'PYEOF'`) with arguments, as done in `compute-run-score.sh` line 113 and `score-reflector.sh` line 111. The `cmd_scores()` function should pipe `score_event` to Python via stdin instead of interpolating it.

**Recommended fix:**
```bash
echo "$score_event" | python3 -c "
import json, sys
event_json = sys.stdin.read()
event = json.loads(event_json)
# ... rest of formatting ...
" 2>/dev/null || { ... }
```

This follows the `cmd_timeline()` pattern and eliminates the interpolation entirely.

### m3: cmd_trend in audit-log-query.sh does not handle non-numeric `--dimension` values safely

**File:** `/Users/imurphy/projects/claude-devkit/scripts/audit-log-query.sh`, `cmd_trend()`, line 690

The `dimension` variable from `--dimension <name>` is interpolated into Python without sanitization:
```bash
dimension = '${dimension}'
```

If a user passes `--dimension "'; import os; os.system('id'); x='`, the Python code would execute arbitrary commands. However, `audit-log-query.sh` is an internal developer tool (not exposed to untrusted input), and valid dimension names (`efficiency`, `security`, `quality`) contain only alphanumeric chars. The practical risk is negligible but the pattern is inconsistent with `compute-run-score.sh` and `score-reflector.sh` which use quoted PYEOF to pass arguments. A future-safety improvement would be to pass `dimension` as a Python argument (`python3 -c "..." "${dimension}"`) and access it via `sys.argv`.

**File:** `/Users/imurphy/projects/claude-devkit/scripts/test-integration.sh`, lines 14-15

However, test numbers jump from 4 to 6 (no test 5) and from 8 to 10 (test 9 is manual). This is a historical artifact from test renumbering. The count is correct but the non-sequential numbering creates mild confusion. Not worth fixing now but worth noting for the next integration test addition.

### m5: CLAUDE.md Quantitative Scoring section does not document the `--help` interface for compute-run-score.sh
